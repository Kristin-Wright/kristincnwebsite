    Unit 1, Outcome 3

    3.4 Describe how you would go about testing digital solutions.

    There are two current industry-standard methodologies I could follow to test digital solutions - Test Driven Design (TDD) and Behaviour Driven Design (BDD - as well as any of their various extensions, such as Acceptance Test Driven Design (ATDD) and Continuous Test Driven Design (CTDD).[1]

    TDD [2] focuses on how a digital solution functions from a developer, or a team of developers’, point of view - the driving question behind this approach is “Is this code correct?”. Developers, usually the author themselves, use unit tests before production to get an idea of how their code is performing at a granular level, so that they can then revisit it repeatedly to write the cleanest code possible.

    Unit tests are defined in this article [3] as “the smallest piece of code that can be logically isolated in a system” such as individual functions, methods or classes - as long as it’s not dependent on a database or crosses systems as this would take too long to test.

    Code is written to pass a specific test, then added to the production code.

    Behavioural Driven Development is a complementary methodology to an agile process - this focuses more on testing a certain feature from a user’s point of view, or ‘outside-in’. [4, 5] The question asked with BDD is: “Is this what we should be testing?“ Tests, or scenarios or user stories, are designed in a “Given-When-Then” format (eg. “Given the user’s bank account is in credit, and they made no withdrawals recently, When they attempt to withdraw an amount less than their card's limit, Then the withdrawal should complete without errors or warnings.”

    Test framework tools (Cucumber, Behat, SpecFlow) and syntaxes like Gherkin (a set of defined grammar rules written in plain English understandable in all areas of a business) make BDD possible.

    Both TDD and BDD approaches focus on incremental iteration and testing, resulting in higher software quality and fewer defects ahead of a product release or update. Following an update, I could additionally use regression testing to ensure that the code still works after a change.

    In the specific case of my website’s build, I tested it by ensuring all links (for example, in the nav bar, or the references in the website footer) lead where they were supposed to and that none are broken - still a work in progress! I also sought feedback from the course tutor to get another user experience perspective; I could also have conducted more granular research into how visitors use specific features by using crowd testing sites such as UserTesting.com.[7]
